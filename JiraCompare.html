01

The Dual-View Problem

Operations teams live in Salesforce. Engineering lives in Jira. When a support case is tied to an engineering ticket, neither side has full visibility. Ops can't see whether engineering is working the ticket. Engineering can't see the revenue impact of letting a ticket go stale.

The Connector solves this by syncing fields in both directions. The same Status, Priority, Assignee, and Ticket Age data is visible and reportable in both platforms — without anyone switching tools.

02

Same Data, Two Dashboards

Both dashboards pull from the same underlying data. A status change in Jira updates the Salesforce dashboard, and vice versa.

SF
Salesforce Dashboard
IT Responsiveness Tracker
  • 4 KPI metrics (total, blocked, resolved, stale)
  • Donut chart — tickets by Jira Status
  • Horizontal bar — aging by Jira Assignee
  • Table — stale tickets (no update in 7+ days)
  • Full aging table sorted by Jira Ticket Age
J
Jira Dashboard
IT Responsiveness Tracker
  • Pie chart — tickets by Status
  • 2D matrix — assignee workload by status
  • Revenue at Risk table — deal value per ticket
  • Table — stale tickets (no update in 7+ days)
  • Created vs. Resolved trend line
  • Full aging table sorted by Created date

The Jira dashboard adds one thing Salesforce can't: the Revenue at Risk column. By syncing Opportunity Amount from Salesforce into a Jira custom field, engineers see the dollar value of every stale ticket. A blocked ticket tied to a $3.1M deal gets attention faster than one with no business context.

Operations sees IT accountability in Salesforce. Engineering sees revenue impact in Jira. Both sides are looking at the same data, updated in real time.

What Syncs Where

The Connector maps these fields bidirectionally. Each field is reportable in both platforms using native tools (Salesforce Report Builder, Jira JQL and dashboard gadgets).

Data PointSalesforce FieldSyncJira Field
Ticket StatusJira_Status__c↔Status
PriorityJira_Priority__c↔Priority
SummaryJira_Summary__c↔Summary
AssigneeJira_Assignee__c→Assignee
Last UpdatedJira_Last_Updated__c→Updated (native)
Created DateJira_Created_Date__c→Created (native)
Target End DateJira_Target_End_Date__c↔Due Date
Ticket AgeJira_Ticket_Age__c (formula)—Calculated via JQL
Deal ValueDeal_Value__c←SF Opportunity Amount

The Deal Value field flows from Salesforce into Jira — the reverse of every other field. This is what makes the Revenue at Risk view possible on the engineering side.

06

Setup Summary

Total one-time setup to get both dashboards running.

1

Salesforce: Create custom fields + map in Connector • ~25 min

10 custom fields on the Case object (Status, Priority, Assignee, Last Updated, Issue Key, Summary, Created Date, Target End Date, Ticket Age formula, Deal Value). Map each one in the Connector binding.

2

Jira: Create custom field + save JQL filters • ~15 min

One custom field (SF Opportunity Amount) mapped from Salesforce Deal Value. Five saved JQL filters for the dashboard gadgets.

3

Enable bidirectional sync • ~5 min

Turn on Auto Push and Auto Pull in the Connector connection settings. Enable for all new associations. Every field change syncs automatically in both directions.

4

Build both dashboards • ~25 min

Salesforce: 4 KPI metrics, donut chart, bar chart, 2 tables. Jira: pie chart, 2D stats matrix, 3 filter results tables, created vs. resolved trend.

Total setup time: ~70 minutes. Once configured, both dashboards update automatically. No recurring maintenance, no manual data entry, no sync scripts to run.
